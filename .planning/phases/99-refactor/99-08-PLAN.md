  <action>
Replace all 14 `self.conn.lock().unwrap()` calls in `src/db/mod.rs` with `self.conn.lock().expect("db connection poisoned")`.

**Why `expect()` instead of `?`:** The Mutex `lock()` returns `LockResult<MutexGuard>` where the `Err` variant indicates the lock was poisoned (another thread panicked while holding it). Propagating this with `?` would change method signatures — but wait, all Db methods ALREADY return `anyhow::Result`. So `?` actually works:

```rust
// Option A: expect — explicit, doesn't change semantics
let conn = self.conn.lock().expect("db connection poisoned");

// Option B: ? — propagates poisoning as error
let conn = self.conn.lock().map_err(|e| anyhow::anyhow!("db connection poisoned: {}", e))?;
```

**Decision: Use `expect()` for now.** Reasoning:
- All methods already return `Result` — callers already handle errors
- Mutex poisoning in a single-user TUI app means something went very wrong — panicking with a clear message is more appropriate than silently propagating
- `?` requires mapping the error type since `PoisonError` doesn't implement `Into<anyhow::Error>` (or does it? `anyhow` has a blanket impl for `std::error::Error`... actually `PoisonError` does implement `Error`)
- Actually, `lock()?.method()` IS cleaner and more idiomatic. Let's check if `?` works directly.

**Revised decision: Use `?` directly.** `PoisonError<T>` implements `std::error::Error` (via `Display + Debug`), and `anyhow` accepts anything implementing `Error + Send + Sync`. So `self.conn.lock()?` works directly with `anyhow::Result`.

**Implementation:**
1. In `src/db/mod.rs`, replace all 14 instances of `self.conn.lock().unwrap()` with `self.conn.lock()?`
2. The method signatures already return `Result<_, anyhow::Error>` — no signature changes needed
3. This is a pure search-and-replace within `src/db/mod.rs` only — no caller changes needed since the return types are identical

**Also check sync.rs for lock().unwrap():**
- `src/sync.rs` lines 150 and 171: `shared_date.lock().unwrap()` — these are in the CLI sync runner and use `Arc<Mutex<String>>` for inter-thread date sharing. These should also use `expect("date lock poisoned")` since they're in a non-Result context (inside async blocks). After decomposition, these are in `src/sync/mod.rs`.
